## Five-Layer Source Verification Gate (MASTER_PROMPT Section 4.7)

For every source in the source-category checklist, walk Layers 1–5:

1. **Official origin verification** — verified_from_url, verification_method
2. **Source category verification** — records_available, search_fields
3. **Data access verification** — access_method, public_access_status, document_access_status
4. **Lead value and source role verification** — source_role (PRIMARY_LEAD_SOURCE / SUPPORTING / ENRICHMENT / REFERENCE_ONLY / BLOCKED / NOT_FOUND), lead_value

Record every result in the source's proof packet (the 18 v5.0.0 fields). The
schema enforces population. A source block missing required proof packet fields
fails Phase 0 validation.

## Product rule — official event source-driven lead intelligence

This framework is an OFFICIAL EVENT SOURCE-DRIVEN lead intelligence system.

LEADS come from EVENT-BASED / DISTRESS-BASED sources (Tier 1):

- Clerk records, recorder records
- Court filings (foreclosure, probate, civil judgments, evictions)
- Foreclosure filings, sheriff sale records
- Tax delinquency events, tax sale events
- Probate records, liens, judgments, lis pendens, recorded notices
- Code enforcement events (when they expose violations / liens / demolition /
  condemnation / nuisance)
- Demolition events, condemnation events

SUPPORTING (Tier 2) — strengthens leads but cannot create them:

- Court case detail pages, document images, auction details, sale status,
  probate metadata, judgment details

ENRICHMENT ONLY (Tier 3, NEVER treated as leads):

- Parcel data, GIS data, CAD data, assessor data
- Owner mailing data, tax roll data, bulk property records
- Valuation, beds, baths, year built, equity estimate, absentee, vacancy

If no verified primary event source is accessible in Harris County (clerk, recorder,
court portal, district clerk, sheriff, tax office, tax collector, trustee sale
portal, foreclosure listing portal, tax lien foreclosure listing, auction
vendor, official vendor portal, or posted notices page), mark it blocked, record
the blocker and next_access_strategy in the source proof packet, and stop after
the Build Eligibility Gate. Clerk and recorder are the most common primary
sources but not the only valid ones. Do NOT compensate by filling the dashboard
with enrichment data. A parcel viewer is not a lead intelligence build.

## Phase 0.5 — Auto-Resolve Blockers (MASTER_PROMPT Section 4.14 — v5.1.0-beta+)

If Phase 0 surfaces blocked, ambiguous, dead, wrong-category, or access-constrained
primary lead sources, run Phase 0.5 BEFORE producing the final verdict.

Phase 0.5 flow:

1. For each source with `verification_confidence` LOW/BLOCKED or `source_role`
   BLOCKED_SOURCE, classify the blocker (`blocker_type`)
2. Technical blockers — auto-resolve. Permission blockers — record requirement and stop attempting.
3. Attempt resolution strategies in the approved order (Section 4.14):
   find_official_vendor_link → replace_homepage_with_record_portal →
   discover_public_search_endpoint → inspect_network_requests →
   discover_hidden_api → use_playwright → use_session_cookie → use_seeded_session →
   (cost-gated) use_captcha_solver / use_stealth_browser / use_residential_proxy →
   (credential-gated) use_operator_login / request_free_account /
   use_paid_subscription_if_operator_provides → manual_operator_assisted_pull →
   standing_records_delivery → public_records_request_last_resort → mark_not_available
4. Record every attempt in `auto_resolve_attempts` array on the source
5. Update source's `auto_resolve_status` and `final_resolution_status`
6. Update top-level `auto_resolve_status` and `final_resolution_status`
7. Recompute `build_verdict` — may upgrade to AUTO_RESOLVED_READY_TO_BUILD,
   PARTIALLY_RESOLVED_BUILDABLE, or downgrade to AUTO_RESOLVE_FAILED

Cost gating: before attempting paid/credential paths (steps 9-14 in the strategy list),
check `estimated_cost_category`. FREE/LOW proceeds. MEDIUM/HIGH/UNKNOWN requires operator approval.

Public records request is never the default. It is a final last resort.

still passed and the config is still written. The framework does not
auto-install `jsonschema`.

## Build Eligibility Gate (MASTER_PROMPT Section 4.10)

At the end of Phase 0 (or Phase 0.5 if blockers were attempted), set the top-level fields in `config/counties/harris_tx.json`:

- `build_verdict` — one of: READY_TO_BUILD, READY_WITH_BLOCKERS, RECON_ONLY,
  WAITING_ON_ACCESS, NOT_BUILDABLE_YET, AUTO_RESOLVED_READY_TO_BUILD,
  PARTIALLY_RESOLVED_BUILDABLE, AUTO_RESOLVE_FAILED
- `build_verdict_reason` — plain English explanation
- `build_verdict_at` — ISO 8601 timestamp
- `auto_resolve_status` — overall county auto-resolve outcome
- `final_resolution_status` — final resolution category

Build Mode does not start unless verdict is READY_TO_BUILD, AUTO_RESOLVED_READY_TO_BUILD,
or PARTIALLY_RESOLVED_BUILDABLE AND the operator explicitly approves.

## Do Not Proceed Matrix (MASTER_PROMPT Section 4.11)

Phase 0 stops (with a clean verdict, not a failure) if any of these fire:

1. No verified primary event source found
2. No primary lead source verified
3. Only enrichment sources found
4. All P0 sources are blocked
5. County config fails schema validation
6. Portal proof missing for required P0 sources
7. Dashboard would contain zero event-based leads
8. Only available data is parcel / GIS / CAD / assessor / tax roll / bulk property
9. Source access requires paid subscription or login AND no operator credential declared
10. System cannot verify public access to records or document images
11. No required P0 source reached verification_confidence HIGH/MEDIUM AND no operator_override
